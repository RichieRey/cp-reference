Arquitectura Check Point Quantum

Antes de tocar un comando o una pantalla, necesitas entender cómo Check Point organiza su infraestructura. Este módulo te da el mapa mental completo del ecosistema.

⏱ ~4 horas 📖 Security Mgmt R81.20 + Beginners Guide 🎯 Conceptual

Los tres componentes fundamentales

Check Point separa la inteligencia (gestión) de la ejecución (inspección). Esto es clave: no es un firewall monolítico, sino una arquitectura distribuida con roles bien definidos.

🛡️
Security Gateway
Servidor dedicado que corre software Check Point para inspeccionar tráfico y aplicar las Security Policies en los recursos de red conectados. Es el que "para" o "deja pasar" los paquetes.
🧠
Security Management Server
Servidor dedicado con el software de gestión. Aquí viven las políticas, los objetos, los logs y la configuración de todos los Gateways. No inspecciona tráfico.
🖥️
SmartConsole
Aplicación GUI de Windows que se conecta al Management Server. Es la interfaz desde la que el administrador configura políticas, objetos, y monitorea el entorno.
📐 Relación entre componentes
El administrador trabaja en SmartConsole → los cambios se guardan en el Management Server → al hacer "Install Policy", el Management Server distribuye la política compilada a los Security Gateways.

Topologías de despliegue

Standalone (todo en uno)

Security Gateway y Security Management Server instalados en el mismo servidor. Solo recomendado para laboratorios o entornos muy pequeños. No se usa en producción porque mezcla roles críticos.

[ SmartConsole ] ──SSH/HTTPS──▶ [ Standalone Server ]
(Gateway + Mgmt en mismo equipo)
eth0: 192.168.1.1 (LAN)
eth1: 203.0.113.1 (Internet)

Distributed (recomendado para producción)

Management Server y Security Gateway en servidores separados. El Management Server puede gestionar múltiples Gateways desde un punto central.

[ SmartConsole ]
│ HTTPS/19009
[ Management Server ] ──SIC (TCP 18191)──▶ [ Gateway 1 ]
──SIC (TCP 18191)──▶ [ Gateway 2 ]
──SIC (TCP 18191)──▶ [ Gateway 3 ]

SIC — Secure Internal Communication

SIC es el mecanismo propietario de Check Point con el que todos los componentes se autentican entre sí usando SSL. Sin SIC establecido, el Management Server no puede distribuir políticas al Gateway.

¿Cómo funciona SIC?
La ICA (Internal Certificate Authority) del Management Server emite certificados. Cuando configuras un Gateway en SmartConsole, introduces una One-Time Password (clave de activación) que usaste durante el First Time Wizard del Gateway. SmartConsole hace click en "Initialize" → se establece el túnel SSL → se intercambian certificados → SIC queda "Trust established".
⚠️ SIC roto = no puedes instalar política
Si ves "Communication Initialization Error" al hacer Initialize, verifica: (1) conectividad de red entre Management y Gateway en puerto TCP 18191, (2) que la One-Time Password coincide en ambos lados, (3) que la hora esté sincronizada (diferencia de más de 5 minutos rompe SIC).

Verificar SIC desde CLI del Management

Expert mode — Management Server
# Verificar estado de SIC con un Gateway específico
[Expert@mgmt]# cpca_client lscert -stat Good -kind SIC

# Reset SIC en el Management (extremo más radical)
[Expert@mgmt]# fwm sic_reset

Licenciamiento

Las licencias en Check Point están vinculadas a la dirección IP del servidor. Esto tiene implicaciones importantes: si cambias la IP de un Management Server, todas las licencias quedan inválidas hasta reactivarlas.

✅ Estados de licencia en SmartConsole
Active = Blade activo con licencia válida.
No License = Blade activo pero sin licencia válida (tráfico puede verse afectado).
About to Expire = Expira en 30 días o menos (7 días para evaluaciones).
Quota Exceeded = Licencia válida pero superó el límite de objetos/usuarios.

Proceso de instalación paso a paso

Fase 1: Arranque desde ISO

SmartConsole — Primera Configuración

SmartConsole es el centro de operaciones. Aquí creas objetos, defines políticas, estableces SIC con el Gateway y publicas cambios. Todo lo que configures aquí fluye hacia los Gateways al instalar la política.

⏱ ~5 horas 📖 Security Management R81.20 + Beginners Guide 🔧 Práctico

Instalación de SmartConsole

SmartConsole es una aplicación Windows. Se descarga directamente desde el Gaia Portal del Management Server.

  1. Abre un navegador y ve a https://<IP-del-Management>. Acepta el certificado auto-firmado.
  2. En el Gaia Portal, busca el enlace "Download SmartConsole" en la página principal.
  3. Instala el ejecutable descargado en tu PC Windows (doble clic, acepta los defaults).
  4. Abre SmartConsole. En la pantalla de login: Server = IP del Management, User = admin, Password = contraseña de admin.
  5. Al conectar por primera vez, SmartConsole mostrará el fingerprint del Management Server. Verifica que coincide con la salida de fwm fingerprint en el Management y haz click en "Proceed".
✅ Verificar fingerprint desde CLI
Expert mode — Management Server
[Expert@mgmt]# fwm fingerprint

Crear el objeto Gateway en SmartConsole

El Gateway ya está instalado con Gaia, pero SmartConsole no lo conoce aún. Debes crear un objeto que lo represente y establecer SIC.

  1. En la vista Gateways & Servers, haz click en New → Gateway.
  2. En la ventana de creación, haz click en "Classic Mode" (no el Wizard).
  3. Completa el campo Name (ej. cp-gw-01) y el campo IPv4 Address (IP de gestión del Gateway).
  4. En la sección Platform, selecciona el Hardware correcto (Open server si es VM), Version (R81.20) y OS (Gaia).
  5. Haz click en Communication. En Platform selecciona "Open server / Appliance". Introduce la Activation Key (la misma que pusiste en el FTCW del Gateway). Click en Initialize.
  6. Si SIC es exitoso, verás "Trust Established". Cierra la ventana.
  7. Haz click en "Get Interfaces" para importar automáticamente las interfaces del Gateway. Revisa la topología (qué interfaces son internas/externas) y ajusta si es necesario.
  8. En la pestaña Network Security, habilita los Blades que necesitas (al menos Firewall).
  9. Haz click en OK. El objeto aparece en la lista de Gateways & Servers.
⚠️ Si SIC falla — checklist
1. ¿Hay conectividad TCP al puerto 18191 entre Management y Gateway? (telnet <gw-ip> 18191)
2. ¿La Activation Key es exactamente la misma? (sensible a mayúsculas)
3. ¿La diferencia horaria es menor a 5 minutos? (show clock en ambos)
4. ¿Los servicios CP están corriendo en el Gateway? (cpwd_admin list)

Sesiones, Publish y Install Policy

Check Point usa un modelo de sesiones. Cuando haces cambios en SmartConsole, no se aplican de inmediato — se guardan en tu sesión local (borrador). Para que sean permanentes y lleguen a los Gateways, debes:

💾
1. Publish
Guarda los cambios de tu sesión en la base de datos del Management Server. Otros administradores pueden verlos ahora. Botón en la barra superior, o Ctrl+S.
📤
2. Install Policy
Compila la política y la distribuye a los Gateways seleccionados. Sin este paso, los cambios no llegan al firewall. Se hace desde el menú Policy → Install Policy.
🔄
Revisiones
Cada Publish crea una revisión. Puedes ver el historial y revertir a una revisión anterior desde Manage & Settings → Revisions.
📌 Flujo completo de un cambio
Modificar regla en SmartConsole → Publish (ctrl+S) → Install Policy → Seleccionar Gateway destino → OK → La política llega al Gateway en segundos.

Reglas de Seguridad y Política de Acceso

El corazón de Check Point. Aquí aprendes a construir una política de firewall correcta: con orden lógico, reglas implícitas, Stealth Rule y las mejores prácticas de diseño.

⏱ ~4 horas 📖 Security Management R81.20 🔧 Práctico

Anatomía de una regla

Cada regla en Check Point tiene estos campos (columnas en la tabla de la política):

CampoDescripciónEjemplo
No.Número de regla. Las reglas se evalúan de arriba a abajo, primera coincidencia gana.1, 2, 3…
NameNombre descriptivo de la regla. Muy importante para auditoría."Acceso-Web-LAN"
SourceOrigen del tráfico. Puede ser Any, un objeto, un grupo.LAN-Corp
DestinationDestino del tráfico.Internet (Any)
Services & ApplicationsPuerto/protocolo o aplicación específica.HTTP, HTTPS
ActionQué hacer con el tráfico que coincide.Accept, Drop, Reject
TrackCómo registrar el evento. Log genera entrada en SmartConsole.Log, None, Alert
Install OnEn qué Gateway(s) instalar esta regla.Policy Targets (todos)

Reglas implícitas — Las que no ves pero actúan

Check Point tiene reglas que se aplican antes y después de tu política, aunque no las veas en la lista. Para verlas: menu Policy → Implied Rules (o Edit → Policy → Global Properties).

Regla implícitaPosiciónAcción
Accept Control ConnectionsAntes de todasAccept — Permite comunicación SIC entre Management y Gateway
Accept ICMP requestsConfigurableAccept — Permite ping al Gateway (deshabilitar en producción)
Drop All (Cleanup Rule)Al final de todoDrop — Todo lo que no haya coincidido con ninguna regla se descarta
⚠️ La Cleanup Rule
La última regla implícita es un DROP sin log. Esto significa que si tu tráfico llega a esta regla, se descarta silenciosamente y sin log. Por eso es buena práctica agregar una Cleanup Rule explícita con Track=Log al final de tu política, para poder ver qué está siendo descartado.

Estructura recomendada de una política

# Orden recomendado de reglas en una política profesional

Regla 1: STEALTH RULE → Drop tráfico AL Gateway mismo (protege el Gateway)
Source: Any | Dest: cp-gw-01 | Action: Drop | Track: Log

Regla 2: Administración → Permite acceso SSH/HTTPS solo desde hosts admin
Source: AdminPCs | Dest: cp-gw-01 | Svc: SSH,HTTPS | Action: Accept

Reglas 3-N: Negocio → Tráfico legítimo de la organización
[De más específico a más general]

Última: CLEANUP RULE → Drop todo lo demás con Log
Source: Any | Dest: Any | Action: Drop | Track: Log

La Stealth Rule — Por qué es crítica

La Stealth Rule protege al Gateway de acceso directo no autorizado. Si alguien intenta conectarse directamente al firewall (SSH, Telnet, HTTP), esta regla lo bloquea antes de que llegue a las reglas de administración legítima.

🚨 Error común de principiantes
Poner la regla de administración antes de la Stealth Rule. El orden correcto es: Stealth Rule primero (Drop al Gateway), luego la excepción para los administradores.

  2. Si no hay política, click en "New Policy". Dale un nombre (ej. "Corporate-Policy"). Selecciona el Gateway como Installation Target.

  5. Para agregar Section Headers: clic derecho en el número de regla → "Add Section Above/Below". Esto organiza visualmente la política sin afectar la evaluación.
  6. Cuando termines: Publish → Install Policy → selecciona el Gateway → OK.

NAT — Network Address Translation

NAT en Check Point tiene dos sabores principales: Automático (configurado en el objeto) y Manual (reglas explícitas en la tabla NAT). Ambos coexisten y tienen precedencias específicas.

⏱ ~4 horas 📖 Security Management R81.20 + sk30197 🔧 Práctico

¿Cómo procesa Check Point las reglas NAT?

El Security Gateway aplica las reglas NAT según este orden estricto. Cuando una conexión coincide con una regla, se aplica y no se siguen evaluando las siguientes (excepto en Automatic NAT donde pueden coincidir dos: una para Source y otra para Destination):

Orden de evaluación NAT (de mayor a menor prioridad):

1. Manual NAT Rules # Primera coincidencia gana, sin mirar más
2. Automatic Static NAT # Para Gateway y Host objects
3. Automatic Hide NAT # Para Gateway y Host objects
4. Automatic Static NAT # Para Network y Address Range objects
5. Automatic Hide NAT # Para Network y Address Range objects
📌 Diferencia clave: Manual vs Automatic NAT
Manual NAT: primera regla coincidente se aplica, se detiene ahí.
Automatic NAT: pueden coincidir DOS reglas simultáneamente — una que traduce el Source y otra que traduce el Destination. Check Point las aplica ambas.

Hide NAT (PAT) — Acceso a Internet desde LAN

El caso más común: múltiples IPs internas se esconden detrás de una IP pública del Gateway. En esencia es PAT (Port Address Translation).

Método 1: Automatic Hide NAT (configurado en el objeto Network)

  1. En SmartConsole, abre el objeto de red interna (ej. LAN-Corp).
  2. Navega a la pestaña NAT.
  3. Marca "Add Automatic Address Translation Rules".
  4. En Translation method, selecciona Hide.
  5. En "Hide behind", selecciona "Gateway" (usa la IP de la interfaz externa del Gateway).
  6. Click OK. SmartConsole genera automáticamente 2 reglas NAT en la tabla NAT:
    • Regla intranet: no traduce tráfico LAN→LAN
    • Regla salida: traduce Source de LAN-Corp → IP externa del Gateway
  7. Publica e instala la política.

Método 2: Hide NAT automático en el Gateway (más simple aún)

  1. Abre el objeto del Security Gateway.
  2. Ve a la sección NAT.
  3. Marca "Hide internal networks behind the Gateway's external IP".
  4. Click OK, publica e instala. Todo tráfico saliente se traduce a la IP externa automáticamente. No se genera ninguna regla visible en la tabla NAT.

Static NAT — Publicar un servidor interno

Cuando necesitas que un servidor interno sea accesible desde Internet con una IP pública fija. La traducción va en ambas direcciones.

Automatic Static NAT (más común)

  1. Abre el objeto Host del servidor interno (ej. WebServer-DMZ con IP 10.0.100.10).
  2. Ve a la pestaña NAT.
  3. Marca "Add Automatic Address Translation Rules".
  4. Translation method: Static.
  5. En "Hide behind IP Address", introduce la IP pública (ej. 203.0.113.10).
  6. Click OK. SmartConsole genera 3 reglas automáticas:
    • Intranet: no traduce tráfico del servidor a sí mismo
    • Salida: Source del WebServer → IP pública
    • Entrada: Dest IP pública → IP privada del WebServer
  7. Verifica también que tengas una regla en la Access Control Policy que permita el tráfico entrante al servidor.
⚠️ No olvides la regla de acceso
NAT solo traduce la IP — no permite el tráfico. Necesitas una regla explícita en la Access Control Policy: Source=Any, Dest=WebServer-DMZ (IP interna), Service=HTTP+HTTPS, Action=Accept.

Manual NAT Rules — Control total

Cuando el NAT automático no es suficiente: necesitas traducir Source Y Destination al mismo tiempo, traducir puertos, o hacer NAT condicional por Source específico.

Configuración de Manual NAT para Static NAT de servidor

Proxy ARP — El paso que muchos olvidan

Para NAT Manual (y en algunos casos de NAT Automático), el Gateway necesita responder a los ARP requests por la IP pública traducida. Esto se llama Proxy ARP.

¿Qué es Proxy ARP?
Cuando un router externo hace ARP Request para 203.0.113.10 (la IP pública del servidor), el Gateway debe responder con su propia MAC diciendo "esa IP está aquí". Si no hay Proxy ARP, el router no sabe a dónde mandar el tráfico y las conexiones entrantes fallan.

Hay dos formas de configurar Proxy ARP para Manual NAT según sk30197:

Opción A: Configuración en Global Properties (método recomendado)

  1. En SmartConsole: Menu → Global Properties → NAT - Network Address Translation.
  2. En la sección "Automatic NAT rules": si "Automatic ARP configuration" está marcado, también marca "Merge manual proxy ARP configuration".
  3. Esto hace que el Gateway procese tanto el ARP automático como el manual (del archivo local.arp).

Opción B: Agregar entrada en Gaia Portal (más visual)

  1. Accede al Gaia Portal del Gateway.
  2. Ve a Network Management → ARP.
  3. En la sección "Proxy ARP", haz click en Add.
  4. En "Proxy ARP Address" ingresa la IP pública (203.0.113.10).
  5. En "Real IP Address" ingresa la IP de la interfaz externa del Gateway (203.0.113.1). En cluster, usa la IP física del miembro, no la VIP.
  6. Click OK y guarda configuración.

Arquitectura de logging

Los Security Gateways generan logs de red. El Management Server genera logs de auditoría (acciones de administradores). Los logs pueden almacenarse en tres lugares distintos:

🖥️
Management Server (default)
Almacena logs de los Gateways gestionados. Válido para entornos pequeños/medianos.
📦

Servidor separado para organizaciones que generan muchos logs. Recomendado para producción. Descarga el Management Server.

El Gateway guarda logs localmente. Solo cuando no hay conectividad con el Log Server o para casos especiales.

El Log Server indexa automáticamente los logs para permitir búsquedas instantáneas sobre millones de registros. Importante: SmartLog Indexing no se habilita por defecto después de upgrade o nueva instalación en servidores Smart-1 205, Smart-1 210, o Open Servers con menos de 4 cores.

Actualizaciones: CPUSE y Jumbo Hotfix Accumulator

Mantener el sistema actualizado es crítico para la seguridad y estabilidad. Check Point usa CPUSE como gestor de paquetes de Gaia y distribuye correcciones agrupadas en Jumbo Hotfix Accumulators (JHF).

⏱ ~4 horas 📖 Gaia Admin Guide + sk92449 🔧 Práctico

Qué es CPUSE

CPUSE (Check Point Upgrade Service Engine) es el gestor de paquetes integrado en Gaia OS. Permite instalar, verificar y desinstalar paquetes de software de Check Point sin necesidad de reinstalar el sistema.

🔧
Hotfix individual
Corrección específica para un bug o vulnerabilidad. Se instala sobre la versión base.
📦
Jumbo HFA (JHF)
Colección de hotfixes agrupados en un solo paquete acumulativo. Cada "Take" incluye todos los anteriores.
⬆️
Major Upgrade
Actualización de versión principal (ej. R81.10 → R81.20). Requiere snapshot previo.

Instalar Jumbo HFA desde Gaia Portal (método recomendado)

  1. Accede al Gaia Portal del Gateway (https://<IP-GW>).
  2. En el árbol izquierdo, navega a Upgrades (CPUSE) → Status and Actions.
  3. Si el Gateway tiene acceso a Internet, haz click en "Check for Updates". Los paquetes disponibles aparecen automáticamente.
  4. Si NO hay Internet (offline): descarga el paquete CPUSE desde el Support Center de Check Point en tu PC. Luego en el Portal haz click en "Import Package" → Browse → selecciona el archivo descargado.
  5. En la lista de paquetes, busca el Jumbo HFA. Haz clic derecho → "Verify Update" primero. Verifica que no haya errores.
  6. Haz clic derecho → "Install". El sistema instala el paquete y reinicia automáticamente.
🚨 NUNCA hagas "Install" cuando querías "Upgrade"
En CPUSE hay dos opciones diferentes: Install hace una instalación limpia (borra toda la configuración y reinicia todos los miembros). Upgrade actualiza preservando la configuración. Para un Jumbo HFA siempre usa Install. Para una actualización de versión mayor usa Upgrade.

Upgrade de versión mayor — Proceso general

Actualizar de R81.10 a R81.20 (o similar) requiere planificación. La secuencia oficial es:

  1. Actualiza primero el Management Server. Siempre debe ir a la misma versión o superior al Gateway. El Management puede gestionar Gateways de versiones anteriores, pero no al revés.
  2. Crea un snapshot del Management Server antes de actualizar (ver Módulo 10).
  3. Ejecuta el Pre-Upgrade Verifier para detectar problemas antes de actualizar.
  4. Instala el paquete de upgrade con CPUSE usando la opción "Upgrade" (no "Install").
  5. Reinicia el servidor. El upgrade aplica los cambios durante el boot.
  6. Actualiza los Gateways uno por uno (o usando SmartUpdate para actualizaciones masivas).
✅ Best Practice: Management primero, siempre
El Management Server debe estar en la versión más reciente o igual a los Gateways. Un Management Server R81.20 puede gestionar Gateways R81.10, R80.40, etc. Un Gateway R81.20 gestionado por Management R81.10 puede generar problemas de compatibilidad.

Backup, Restore y Snapshots de Gaia

Antes de cualquier cambio importante — upgrade, modificación de configuración crítica, cambio de IPs — debes tener un backup. Check Point ofrece tres métodos distintos con propósitos diferentes.

⏱ ~3 horas 📖 Gaia Admin Guide + Security Management R81.20 🔧 Práctico

Los tres métodos de backup

No son equivalentes. Cada uno sirve para un propósito diferente y debes conocer las diferencias:

MétodoQué incluyeCuándo usarDónde se guarda
Gaia SnapshotImagen completa del OS + todas las configuraciones de Gaia + paquetes instaladosAntes de upgrade de versión mayor. Permite rollback completo del sistema.Partición local de Gaia. SCP para transferir afuera.

Gaia Snapshot — Backup completo del sistema

Crear snapshot desde Gaia Portal

  1. Accede al Gaia Portal → Maintenance → System Backup.
  2. En la sección Snapshots, haz click en "New Snapshot".
  3. Dale un nombre descriptivo con fecha: pre-upgrade-20260416.
  4. Click OK. El proceso puede tardar varios minutos dependiendo del tamaño.

Crear snapshot desde Clish

Gaia Clish — Snapshots
# Crear snapshot con nombre descriptivo
hostname> add snapshot name pre-jhf-$(date +%Y%m%d) description "Before JHF Take 119"

# Ver snapshots disponibles
hostname> show snapshots
Name Created Description
pre-jhf-20260416 Apr 16 2026 10:00 Before JHF Take 119

# Restaurar snapshot (revierte TODO el sistema al estado del snapshot)
hostname> set snapshot revert name pre-jhf-20260416

# Exportar snapshot a servidor remoto por SCP
hostname> set snapshot export name pre-jhf-20260416 target-url scp://admin@10.0.0.50/backups/gw-snapshot.tgz

# Eliminar snapshot antiguo para liberar espacio
hostname> delete snapshot name old-snapshot-20250101
⚠️ Snapshots ocupan espacio significativo
Un snapshot es una imagen completa del sistema. Puede ocupar 10-30 GB dependiendo de la instalación. Verifica el espacio disponible antes de crear uno: df -h en Expert mode. Exporta los snapshots a un servidor externo si el disco local es limitado.

Backup de la base de datos del Management — migrate_server

Este es el backup más importante para el Management Server. Contiene todas las políticas, objetos, usuarios, certificados y configuración de administración.

Expert mode — Management Server — migrate_server
# Primero navegar al directorio de scripts
[Expert@mgmt:0]# cd $FWDIR/scripts/

# Verificar que la exportación es posible (Pre-Upgrade Verifier)
[Expert@mgmt:0]# ./migrate_server verify -v R81.20

# Exportar la base de datos del Management
[Expert@mgmt:0]# ./migrate_server export -v R81.20 /var/log/mgmt-backup-20260416.tgz

# Importar en el mismo servidor (rollback) o en nuevo servidor
[Expert@mgmt:0]# ./migrate_server import -v R81.20 /var/log/mgmt-backup-20260416.tgz

# Ver log del proceso de migración
[Expert@mgmt:0]# ls /var/log/opt/CPshrd-R81.20/migrate-*.log
🚨 Compatibilidad de versiones en migrate_server
Si vas a importar el backup en un Management Server con versión diferente (upgrade), debes usar las herramientas de migración del paquete creado específicamente para esa versión destino — no el migrate_server de la versión origen.

Si origen y destino son la misma versión, también deben tener el mismo Jumbo HFA Take instalado. Verifica con cpinfo -y all en ambos servidores.

Recuperación ante desastres — Management HA

Si tienes Management en Alta Disponibilidad (Primary + Secondary) y el Primary falla permanentemente, el proceso de promoción del Secondary es:

  1. En el servidor Secondary, detén los servicios: cpstop.
  2. Elimina los archivos de estado de Secondary: los archivos $FWDIR/state/ha_secondary*. Estos contienen la información del estado Secondary — al eliminarlos, el servidor arrancará como Primary.
  3. Verifica que tienes una licencia mgmtha en el servidor que será promovido. Recuerda: las licencias están vinculadas a la IP.
  4. Ejecuta cpstart en el servidor promovido.
  5. En SmartConsole: elimina todas las instancias del objeto del Management antiguo (click derecho → "Where Used" para encontrarlos todos).
  6. Instala la base de datos: Menu → Policy → Install Database.
  7. Instala un nuevo Secondary Management Server con la IP del antiguo Primary.
  8. Resetea SIC y reconecta el nuevo Secondary.

Smart-1 Cloud — Gestión en la Nube

Smart-1 Cloud es la alternativa SaaS al Management Server on-premises. Cero mantenimiento, escalabilidad automática, siempre en la última versión. Accesible desde el Infinity Portal de Check Point.

⏱ ~3 horas 📖 Smart-1 Cloud Admin Guide ☁️ Cloud

¿Qué es Smart-1 Cloud?

Smart-1 Cloud es el Management Server de Check Point como servicio en la nube. Elimina la necesidad de mantener, actualizar y hacer backup del servidor de gestión. Check Point lo gestiona por ti.

⚡
Cero mantenimiento
Check Point aplica las actualizaciones, hotfixes y backups automáticamente. Nunca más administrar el Management Server.
📊
Infinity Portal
Interfaz web centralizada desde la que se accede a Smart-1 Cloud, SD-WAN, IoT, y otros servicios de la plataforma Infinity.
🔗
Mismo SmartConsole
Se usa exactamente el mismo SmartConsole que con on-premises. Solo cambia el método de login (token en lugar de IP+password).
🌐
Multi-entorno
Gestiona Gateways on-premises, CloudGuard (AWS/Azure/GCP), Maestro y Quantum Spark desde una sola consola.

Primeros pasos con Smart-1 Cloud

  1. Crear cuenta en Infinity Portal: ve a portal.checkpoint.com y crea una cuenta. Si ya tienes contrato, vincúlala a tu User Center account para que las licencias se carguen automáticamente.
  2. Acceder a Smart-1 Cloud: en el Infinity Portal, click en el menú ☰ (esquina superior izquierda) → Quantum → Security Management & Smart-1 Cloud.
  3. Crear el servicio: si es la primera vez, click en "Create a new Smart-1 Cloud Management". Configura:
    • Service Identifier: prefijo para el identificador único del servicio
    • Desired Log Retention Period: cuántos días guardar los logs. Los logs más antiguos se eliminan automáticamente cuando el almacenamiento se llena.
  4. Click Create. Espera 1-2 minutos. Recibirás un email de confirmación.
  5. Descargar SmartConsole desde el portal: Welcome → Download SmartConsole for Windows. Hay dos versiones: Installation (instalador completo) y Portable (ZIP sin instalación).

Conectar SmartConsole a Smart-1 Cloud

El login con Smart-1 Cloud es diferente al login tradicional (no es IP + user/password):

  1. Abre SmartConsole. En el campo Server, selecciona "Cloud" del dropdown.
  2. Introduce el Management Connection Token. Este token se obtiene en Smart-1 Cloud Portal → Settings → SmartConsole → icono de copiar.
  3. Click "Infinity Login". SmartConsole se cierra y abre el navegador.
  4. En el navegador, introduce tus credenciales del Infinity Portal.
  5. SmartConsole se reabre ya autenticado y conectado a la instancia en la nube.

Conectar Gateways on-premises a Smart-1 Cloud

  1. En SmartConsole (ya conectado a Smart-1 Cloud) → Gateways & Servers → New → Gateway.
  2. Sigue el proceso normal de creación de objeto Gateway.
  3. En el paso de SIC, el Gateway necesita conectividad saliente a Internet hacia los servidores de Smart-1 Cloud (puertos HTTPS/443).
  4. Si el Gateway corre versión R80.40 con JHF Take 89 o superior (o R81.10+), no se requiere reset de SIC. Para versiones anteriores, sí hay que resetear SIC antes (sk65764).

Migrar desde Management on-premises a Smart-1 Cloud

Si ya tienes un Management Server local y quieres migrar a Smart-1 Cloud, hay dos escenarios:

Escenario A: Management conectado al Infinity Portal

  1. En Infinity Portal, selecciona el Management Server conectado.
  2. Click en el menú de tres puntos → "Run Pre-migrate verifier" para verificar compatibilidad.
  3. Click "Migrate to Smart-1 Cloud". El proceso puede tardar bastante tiempo. El servicio estará no disponible durante la importación.
  4. Recibirás un email cuando complete. Luego conecta los Gateways.

Escenario B: Management NO conectado al Infinity Portal

  1. En Smart-1 Cloud Portal → Settings → Migrate → Download las herramientas de migración.
  2. En el Management Server local, ejecuta la herramienta de exportación. Se genera un archivo de migración.
  3. En Smart-1 Cloud Portal → Import and Start → Upload & Start. Sube el archivo.
🚨 Limitaciones importantes de Smart-1 Cloud
No soportado: Management High Availability (HA), Multi-Domain Security Management (MDS), Log Server dedicado.
Al migrar un entorno Standalone: se separa automáticamente en Management y Gateway (cambio permanente).
Al migrar un entorno HA: debes eliminar el Secondary Management después de migrar.

Multi-Domain Security Management (MDS)

Para proveedores de servicio (MSSP) o grandes empresas con múltiples unidades de negocio independientes, MDS permite aislar entornos de gestión completos dentro de un mismo servidor físico.

⏱ ~4 horas 📖 Security Management R81.20 🏢 Empresarial

Arquitectura Multi-Domain

Arquitectura MDS completa:

[ Multi-Domain Server (MDS) ] ← Servidor físico central
│
├──▶ [ Domain Management Server: Cliente A ]
│ └──▶ Gateways de Cliente A
│
├──▶ [ Domain Management Server: Cliente B ]
│ └──▶ Gateways de Cliente B
│
└──▶ [ Domain Management Server: Cliente C ]
└──▶ Gateways de Cliente C

Cada dominio es completamente aislado: políticas, objetos, logs, administradores.

Componentes clave

ComponenteDescripción
Multi-Domain Server (MDS)Servidor físico que aloja todos los Domain Management Servers virtuales. El contenedor principal.
Domain Management Server (DMS)Instancia virtual de Management Server dentro del MDS. Cada cliente/dominio tiene el suyo, completamente aislado.
Global DomainDominio especial que contiene objetos y políticas compartidas entre todos los dominios.
Global PolicyPolítica corporativa base aplicada a todos los dominios. Los administradores de dominio no pueden modificarla.
Multi-Domain Log Server (MDLS)Servidor dedicado para centralizar logs de todos los dominios. Tiene Domain Log Servers por dominio.
Superuser (Global Admin)Administrador con acceso a todos los dominios y al Global Domain.
Domain AdminAdministrador con acceso solo a su dominio. No puede ver ni modificar otros dominios.

Instalación de MDS

Durante el First Time Configuration Wizard de Gaia, en la ventana de Installation Type, selecciona "Multi-Domain Server" en lugar de "Security Gateway and/or Security Management". Esto instala el rol de MDS.

⚠️ MDS no soporta IPv6
Multi-Domain Server no soporta IPv6 en absoluto (Known Limitation PMTR-14989 oficial de Check Point). Toda la infraestructura MDS debe ser IPv4.

Administración desde SmartConsole con MDS

Al conectar SmartConsole a un MDS, el flujo de trabajo es diferente:

  1. Al conectar, SmartConsole muestra la pantalla del MDS con la lista de todos los dominios configurados.
  2. Selecciona el dominio al que quieres acceder. SmartConsole "entra" en ese dominio — es como conectarse a un Management Server independiente.
  3. Para cambiar de dominio: haz click en el nombre del dominio en la barra superior → selecciona otro dominio.
  4. Para acceder al Global Domain y las Global Policies: selecciona "Global" en la lista de dominios.

Global Policies — Estándares corporativos

Las Global Policies permiten definir reglas de seguridad obligatorias que se aplican a todos los dominios. Los administradores de dominio las ven pero no pueden modificarlas ni eliminarlas.

Caso de uso típico
Una empresa tiene 10 subsidiarias con sus propios equipos de seguridad. El equipo central define una Global Policy con la Stealth Rule corporativa, bloqueo de países de alto riesgo y reglas de compliance PCI. Cada subsidiaria tiene su propio DMS y administradores, pero no pueden saltarse las reglas globales.

Identity Awareness — Control por Identidad

Identity Awareness transforma las políticas de firewall: en lugar de basarse solo en IPs, permite controlar el acceso por usuario, grupo de Active Directory y equipo. El Gateway "sabe" quién está detrás de cada IP.

⏱ ~6 horas 📖 Identity Awareness R81.20 🔧 Práctico

Fuentes de identidad disponibles

Identity Awareness puede obtener la identidad del usuario desde múltiples fuentes. La elección depende del entorno y el nivel de seguridad requerido:

FuenteCómo funcionaCuándo usarla
AD QueryConsulta los Security Event Logs del Active Directory vía WMI. Completamente transparente — sin agente en el cliente. Detecta usuarios cuando se autentican en el dominio.Mayoría de entornos corporativos con AD. Logging básico con mínima configuración.
Browser-Based Auth (Captive Portal)Redirige al usuario a un portal web donde introduce sus credenciales. Útil para dispositivos no Windows (iPads, Linux, invitados).Usuarios no identificables por AD Query: móviles, Linux, contratistas.
Transparent Kerberos AuthEl browser obtiene un ticket Kerberos de AD automáticamente (SSO). Sin input del usuario si está en el dominio.Entornos con SSO corporativo. Se configura como fallback al Captive Portal si falla.
Identity CollectorAgente ligero que recolecta identidades de múltiples fuentes y las envía al Gateway.Entornos complejos con múltiples fuentes de identidad.
RADIUS AccountingEl Gateway intercepta los mensajes RADIUS Accounting Start/Stop para mapear usuario→IP.Usuarios VPN o WiFi con autenticación RADIUS.
Remote AccessIdentifica usuarios que se conectan por VPN con Mobile Access o IPsec en Office Mode.Teletrabajadores con VPN corporativa.

AD Query — El método más común

AD Query es la fuente más usada por su sencillez. No requiere ningún software en los clientes. Se basa en Windows Management Instrumentation (WMI) para consultar el Security Event Log del Domain Controller.

¿Cómo funciona AD Query?
Cuando un usuario Windows hace login en el dominio, el Domain Controller genera un evento en el Security Event Log (Event ID 4768 o 4769 — TGT/TGS Kerberos ticket). El Gateway consulta periódicamente estos eventos vía WMI y extrae el mapeo usuario → IP del equipo. El usuario no nota nada.
⚠️ Requisito de credenciales para AD Query
AD Query requiere credenciales de administrador de dominio para leer el Security Event Log. Browser-Based Authentication solo necesita credenciales estándar. Impacto en Domain Controller: menos del 3% de CPU adicional por cada 1000 usuarios.

Arquitectura PDP / PEP — Identity Sharing

En entornos con múltiples Gateways, no es eficiente que cada Gateway consulte el AD por separado. Identity Sharing resuelve esto:

Sin Identity Sharing (ineficiente):
[GW-1] ──WMI──▶ [AD DC] ← cada GW consulta por separado
[GW-2] ──WMI──▶ [AD DC]
[GW-3] ──WMI──▶ [AD DC]

Con Identity Sharing (eficiente):
[GW-1 = PDP] ──WMI──▶ [AD DC] ← solo el PDP consulta al AD
│ Identity Sharing (Push/Smart-Pull)
├──▶ [GW-2 = PEP] ← recibe identidades del PDP
└──▶ [GW-3 = PEP]
RolFunción
PDP (Policy Decision Point)Obtiene identidades de las fuentes (AD, Captive Portal, etc.) y las comparte con los PEPs. Solo uno o pocos Gateways necesitan ser PDP.
PEP (Policy Enforcement Point)Recibe identidades del PDP y las usa para aplicar las Access Roles en su política. No consulta al AD directamente.

Configurar Identity Awareness — Paso a paso

Paso 1: Habilitar el blade en el Gateway

  1. En SmartConsole → Gateways & Servers → doble click en el Gateway.
  2. En la pestaña Network Security, marca "Identity Awareness". El wizard de configuración se abre automáticamente.

VPN Site-to-Site — IPsec / IKE

La VPN Site-to-Site crea túneles cifrados entre sedes. Check Point usa el estándar IPsec con IKE para negociación de claves. El modelo de configuración es centralizado en SmartConsole mediante VPN Communities.

⏱ ~8 horas 📖 Site-to-Site VPN R81.10 🔧 Práctico

Conceptos fundamentales

🔑
IKE (Internet Key Exchange)
Protocolo de gestión de claves que crea los túneles VPN. Negocia los parámetros de cifrado y autentica los extremos. Versiones: IKEv1 e IKEv2.
🔒
IPsec
Protocolo de cifrado que protege el tráfico una vez negociado el túnel IKE. Garantiza: Autenticidad, Privacidad (cifrado) e Integridad.
🌐
VPN Domain
Colección de redes internas de un Gateway que viajan cifradas por el túnel. Define qué tráfico entra en el túnel VPN.
👥
VPN Community
Colección de VPN Domains y sus atributos. Agrupa los Gateways participantes y define los algoritmos de cifrado compartidos.

Topologías: Star vs Mesh

Mesh Community — todos conectan con todos:
[GW-Londres] ↔ [GW-NYC]
↕ ↕
[GW-Madrid] ↔ [GW-Paris]

Star Community — satélites solo conectan al centro (hub-and-spoke):
[Sucursal-A]
│
[Sucursal-C] ─────▶ [HQ-Central] ◀───── [Sucursal-B]
│
[Sucursal-D]
# Las sucursales NO se comunican entre sí directamente

Configurar VPN entre dos Gateways Check Point (mismo Management)

  1. Habilitar el blade IPsec VPN en ambos Gateways: en el objeto del Gateway → pestaña Network Security → marca "IPsec VPN". Al habilitarlo, se crea automáticamente un certificado ICA para ese Gateway.
  2. Definir el VPN Domain de cada Gateway: en el objeto → Network Management → VPN Domain. Para el Gateway central: selecciona la red interna específica. Para satélites: normalmente "All IP addresses behind the gateway".
  3. Crear la VPN Community: Security Policies → Access Tools → VPN Communities → New → Star Community (o Meshed).
  4. En la Star Community, pestaña Gateways: añade el Gateway HQ como Center Gateway y las sucursales como Satellite Gateways. Activa "Mesh center gateways" si los centros necesitan comunicarse entre sí.
  5. En la pestaña Encryption: configura los algoritmos IKE (Fase 1) e IPsec (Fase 2). Los defaults de Check Point son seguros para la mayoría de casos.
  6. En la pestaña Encrypted Traffic: selecciona "Accept all encrypted traffic" si quieres permitir todo el tráfico VPN sin reglas adicionales, o configura reglas específicas en la Access Control Policy.
  7. En la pestaña VPN Routing (solo Star): selecciona el modo de enrutamiento:
    • To center only — tráfico solo entre satélite y centro
    • To center and to other satellites through center — satélites se comunican entre sí vía el centro
    • To center, or through center to other satellites, to internet and other VPN targets — todo el tráfico incluso Internet va por el centro (VPN Routing para inspección centralizada)
  8. Click OK. Publica e instala la política en todos los Gateways de la comunidad.

VPN con Gateways de terceros (Interop Device)

Para VPN con firewalls Cisco, Fortinet, Palo Alto u otros, el proceso es diferente porque no comparten la misma ICA:

  1. En SmartConsole → Objects → New → Network Object → More → Interoperable Device. Introduce la IP pública del Gateway externo.
  2. Define la topología del Interop Device: Network Management → Topology → define las redes internas del peer externo manualmente.
  3. Define el VPN Domain del Interop Device: las redes que deben viajar cifradas hacia ese peer.
  4. Crea la VPN Community e incluye tu Gateway Check Point como Center y el Interop Device como Satellite (o en Mesh según el caso).
  5. En la Community → pestaña Encryption: configura los mismos parámetros IKE que tiene configurado el administrador del peer externo. Deben coincidir exactamente.
  6. En la Community → pestaña Shared Secret: si se usa Pre-Shared Key (más común con terceros), introduce la PSK acordada con el administrador del peer.
  7. Configura las reglas de acceso y publíca e instala.
✅ Certificados vs PSK
Usar certificados PKI es más seguro que Pre-Shared Keys. Para VPN entre Gateways Check Point del mismo Management, los certificados se gestionan automáticamente vía ICA — no necesitas configurar nada extra. Para Gateways de terceros, se usa habitualmente PSK por simplicidad.

Diagnóstico con fw monitor y zdebug

fw monitor es la herramienta de captura de paquetes a nivel kernel de Check Point. Permite ver el paquete en cada punto del procesamiento interno del firewall, revelando exactamente dónde y por qué se descarta o modifica.

⏱ ~6 horas 📖 Security Management R81.20 + SK30583 🔬 Diagnóstico

Los 4 puntos de inspección del kernel

A diferencia de tcpdump (que captura en la interfaz), fw monitor captura en 4 puntos dentro del procesamiento del kernel del firewall:

Flujo de un paquete entrante (ej. tráfico desde Internet):

[Red Externa]
│
▼
[NIC — eth1]
│
▼ PUNTO i (pre-inbound) ← antes del procesamiento del FW kernel
│
▼ PUNTO I (post-inbound) ← después del procesamiento, antes de ruteo
│ [aquí el FW aplica las reglas de política]
│ [aquí ocurre NAT de Destination si aplica]
│
▼ PUNTO O (pre-outbound) ← antes de salir por la interfaz interna
│
▼ PUNTO o (post-outbound)← en la interfaz interna, ya va a salir
│
[NIC — eth0]
│
[Red Interna]
Cómo leer los puntos de captura
i (minúscula) = Pre-inbound: el paquete llega a la NIC, antes que el FW lo procese.
I (mayúscula) = Post-inbound: el FW lo procesó, incluyendo NAT de destino.
O (mayúscula) = Pre-outbound: va a salir, ya se aplicó NAT de origen.
o (minúscula) = Post-outbound: sale por la NIC hacia el destino.

Si el paquete aparece en i pero no en I → el FW kernel lo descartó (regla Drop, Anti-Spoofing, etc.).
Si aparece en I pero no en O → problema de ruteo o descarte post-inspección.

Sintaxis de fw monitor

Expert mode — fw monitor

[Expert@gw:0]# fw monitor -e 'accept host(10.0.0.1) and host(8.8.8.8);' -o /tmp/test.pcap

# Filtrar por IP y puerto destino (HTTPS)
[Expert@gw:0]# fw monitor -e 'accept src=10.0.0.1 and dport=443;' -o /tmp/https.pcap

# Ver solo en puntos específicos (ej: solo inbound)
[Expert@gw:0]# fw monitor -m iI -e 'accept host(10.0.0.1);'

# NUNCA en producción sin filtro — puede saturar el kernel
# fw monitor sin filtro = captura TODO = peligroso en GW de producción

Protocolo de diagnóstico jerárquico

Antes de usar fw monitor o zdebug, siempre sigue este orden. Los problemas se resuelven el 80% de las veces en los primeros dos niveles:

  1. Nivel 1 — SmartConsole: Verifica que la política está instalada (fw stat). Revisa que el objeto Gateway existe, tiene SIC establecida, y las interfaces tienen la topología correcta (interna/externa). Verifica que no hay anti-spoofing bloqueando.
  2. Nivel 2 — SmartLog / Logs: Busca el tráfico en los logs. ¿Aparece? ¿Qué blade y regla lo descartó? Busca en la columna "Blade" y "Access Rule Name". Un Drop por Cleanup Rule es muy diferente a un Drop por Anti-Bot.
  3. Nivel 3 — CLI: Solo si los niveles 1 y 2 no revelan la causa. Ejecuta fw ctl zdebug + drop primero. Si necesitas ver el paquete en el kernel, usa fw monitor con filtro estricto.
🚨 Precaución con fw monitor en producción
fw monitor sin filtro captura TODO el tráfico del Gateway. En un Gateway de producción con alto throughput, puede generar una carga enorme en el CPU y potencialmente afectar el servicio. Siempre usa filtros específicos (-e 'accept src=...') y limita la duración de la captura. En entornos críticos, coordina una ventana de mantenimiento.

HTTPS Inspection — Inspección SSL/TLS

Más del 90% del tráfico web hoy es HTTPS. Sin inspección SSL, todos los blades de seguridad (IPS, Anti-Bot, Application Control) son ciegos a ese tráfico. HTTPS Inspection resuelve esto con un proxy SSL man-in-the-middle controlado.

⏱ ~4 horas 📖 Security Management R81.20 🔑 SSL/TLS

Cómo funciona HTTPS Inspection

Sin HTTPS Inspection (cifrado opaco):
[Cliente] ──HTTPS(cifrado)──▶ [GW ← no puede inspeccionar] ──▶ [Servidor]

Con HTTPS Inspection (MITM controlado):
[Cliente] ──TLS con cert CA interno──▶ [GW inspecciona] ──TLS con cert real del servidor──▶ [Servidor]
# El GW descifra, inspecciona con todos los blades, y re-cifra

El Gateway actúa como intermediario SSL. Crea dos sesiones TLS separadas: una con el cliente (usando el certificado de la CA interna de Check Point) y otra con el servidor (usando el certificado real del servidor). El tráfico en el Gateway viaja descifrado y es visible para IPS, Anti-Bot, Application Control, etc.

Acciones: Inspect vs Bypass

AcciónQué haceCuándo usar
InspectDescifra el tráfico, lo inspecciona con los blades activos, y lo re-cifra.Tráfico general de navegación web, aplicaciones de negocio, descargas.
BypassNo descifra el tráfico. Pasa tal cual sin inspección SSL.Sitios financieros (bancos), medical, sitios que fallan con MITM (certificate pinning), sitios de privacidad legal.

📌 Comportamiento importante de HTTPS Inspection
Si una regla define Action=Inspect pero el blade configurado en esa regla NO está habilitado en el Gateway específico, la acción real en ese Gateway será Bypass (no Inspect). El log mostrará la regla de Inspect pero la acción efectiva como "Bypass". Esto es diseño intencional documentado en la guía oficial.

Configurar HTTPS Inspection paso a paso

  1. En SmartConsole → Security Policies → HTTPS Inspection → Policy.
  2. Configura el certificado CA que firmará las conexiones inspeccionadas: usa la ICA interna de Check Point (auto-generada) o un certificado de tu PKI corporativa. Ve a HTTPS Inspection → Settings para configurarlo.
  3. Distribuye el certificado raíz a los clientes: si usas la ICA de Check Point, los clientes no confiarán en el certificado por defecto. Necesitas desplegarlo por GPO (Group Policy en AD) o MDM en todos los dispositivos.
  4. Crea las reglas de Bypass primero (sitios financieros, médicos, etc.) para que tengan precedencia.
  5. Crea la regla de Inspect general.
  6. Publica e instala la política.
⚠️ Consideraciones legales y de privacidad
HTTPS Inspection descifra el tráfico personal de los empleados. Antes de activarlo en producción:
1. Consulta con el área legal sobre regulaciones locales (GDPR, privacidad laboral).
2. Informa a los usuarios que su tráfico HTTPS será inspeccionado.
3. Configura reglas de Bypass para sitios de salud, banco personal, y otros donde la inspección podría ser problemática.
4. Considera el impacto en rendimiento: HTTPS Inspection tiene un costo en CPU por el doble de operaciones TLS.

ClusterXL — Alta Disponibilidad de Gateway

ClusterXL agrupa 2 o más Security Gateways en un clúster que trabaja como unidad. Si un miembro falla, el otro toma el tráfico sin interrumpir las sesiones activas. Es la base de cualquier arquitectura de producción.

⏱ ~8 horas 📖 Security Management R81.20 🔧 Práctico

Modos de operación ClusterXL

⚖️
High Availability (Active/Standby)
Un miembro es Active y procesa todo el tráfico. El otro es Standby en espera. En failover, el Standby toma el rol Active. El estándar para producción.
⚡
Load Sharing Unicast
Ambos miembros procesan tráfico simultáneamente. El switch envía tráfico a la VIP MAC (un solo miembro actúa de Pivot y distribuye). Dobla el throughput.
🌐
Load Sharing Multicast
El switch replica el tráfico a todos los miembros vía multicast. Requiere soporte de multicast en los switches. Poco común.

Arquitectura y componentes

Configurar ClusterXL en SmartConsole

  1. Instala y configura Gaia en ambos miembros. Cada miembro debe tener SIC establecida con el Management Server independientemente.
  2. En SmartConsole → Gateways & Servers → New → Cluster → Cluster Wizard (o Classic Mode).
  3. En Classic Mode: introduce el nombre del clúster. En la sección Cluster Members, agrega ambos miembros con sus IPs de management reales.
  4. Para cada miembro, establece SIC con su Activation Key individual.
  5. En la sección Network Topology: para cada interfaz, define las IPs de los miembros y la Cluster VIP. Marca la interfaz de Sync como "Sync".
  6. En la pestaña ClusterXL and VRRP: selecciona el modo (High Availability es el más común).
  7. Selecciona los Software Blades a habilitar. Click OK.
  8. Publica e instala la política en el objeto Cluster.
⚠️ Interfaz de Sync dedicada — obligatorio
La interfaz de sincronización NUNCA debe compartir tráfico de producción. Usa un cable directo entre los miembros (crossover o switch dedicado). Si la sincronización comparte interfaz con tráfico de usuarios, puede ocurrir split-brain y comportamientos impredecibles. En producción, se recomienda Bond en la Sync para mayor redundancia.

SecureXL y CoreXL — Aceleración del Gateway

SecureXL y CoreXL son las tecnologías de rendimiento de Check Point. SecureXL acelera conexiones "conocidas" bypaseando inspección completa. CoreXL distribuye el tráfico entre múltiples cores de CPU en paralelo.

⏱ ~5 horas 📖 Gaia Admin Guide + SK docs ⚡ Performance

Los tres caminos del paquete (Paths)

Todo paquete que entra al Gateway sigue uno de tres caminos, de más rápido a más lento:

Paquete entrante en el Gateway:

[NIC] ──▶ [SND — Secure Network Distributor]
│
┌───────────────────┼────────────────────────────┐
▼ ▼ ▼
[Fast Path / SXL] [Medium Path / PXL] [Slow Path / F2F]
Acelerado SecureXL Aceleración parcial Firewall completo (FWK)
No inspección full Inspección limitada Inspección stateful full
≈ 80-90% del tráfico ≈ 5-10% ≈ 5-10%
│ │ │
└───────────────────┴────────────────────────────┘
│
[NIC de salida]
PathDescripciónCuándo se usa
Fast Path (SXL)SecureXL procesa el paquete directamente sin pasar por el kernel del firewall. Máximo throughput y mínima latencia.Conexiones ya establecidas y clasificadas como "seguras" (templates Accept, Drop, NAT).
Medium Path (PXL)Procesamiento parcial acelerado. Algunos blades inspeccionan pero de forma optimizada.Conexiones que requieren algo de inspección pero no la completa.
Slow Path (F2F)Firewall to Firewall — inspección stateful completa por el kernel del firewall (FWK). El camino "normal" del firewall tradicional.Primera conexión de cada sesión (hasta que se crea el template), conexiones especiales, IPS activo.

SecureXL — Aceleración de conexiones

SecureXL crea templates de aceleración para flujos de tráfico conocidos. Cuando el primer paquete de una conexión pasa por el Slow Path y es aceptado, SecureXL crea un template que permite que los paquetes siguientes de esa misma sesión vayan por el Fast Path directamente.

CoreXL — Múltiples instancias del Firewall kernel

CoreXL replica el kernel del firewall en múltiples instancias (FWK — Firewall Worker), una por CPU core. El SND (Secure Network Distributor) distribuye el tráfico entre las instancias basándose en Source IP + Destination IP + Protocol.

Cómo funciona CoreXL SND
El SND es la parte de CoreXL responsable de: procesar el tráfico entrante de las NICs, acelerar paquetes autorizados (si SecureXL está activo), y distribuir los paquetes no acelerados entre las instancias FWK. La distribución es estática basada en la tupla (src IP, dst IP, protocol) — todas las conexiones entre el mismo par de IPs van siempre a la misma instancia FWK.

Quantum SD-WAN

SD-WAN integra inteligencia de enrutamiento WAN en el Security Gateway. Selecciona automáticamente el mejor enlace ISP para cada aplicación basándose en latencia, jitter, pérdida de paquetes y utilización del enlace.

TipoDescripciónCaso de uso

Criterios de selección de WAN Link

El Gateway monitoriza continuamente cada WAN Link enviando probes ICMP y mide:

⏱️
Latencia
Tiempo de ida y vuelta (RTT) de los probes ICMP. Umbral máximo configurable. Si lo supera, el link se descalifica para ese Steering Behavior.
📉
Jitter
Variación de latencia. Crítico para VoIP y videoconferencia. Calculado sobre los últimos 100 probes.
📦
Packet Loss
Porcentaje de paquetes perdidos. Calculado sobre los últimos 100 probes. Un link con 5%+ de pérdida es inaceptable para la mayoría de aplicaciones.
📊
WAN Utilization
Uso del enlace en %. Si un link está saturado al 90%, se puede excluir automáticamente del pool de selección.

  4. SD-WAN Wizard: en el Infinity Portal → SD-WAN → usa el Wizard que crea automáticamente los objetos dinámicos y reglas base (Overlay para tráfico interno, Default Breakout para Internet).
  5. Configura Steering Behaviors: en Infinity Portal → SD-WAN → Profiles → crea objetos de Steering con los criterios de SLA y tipos de conexión.
  6. Política SD-WAN: define reglas en Infinity Portal → SD-WAN → Policy con Source, Destination, App y el Steering Behavior. Click Publish & Enforce.

Failover manual y Disaster Recovery

Si el Primary falla permanentemente, sigue este proceso para promover el Secondary:

  1. En el Secondary (ahora el único disponible), conecta por SSH. Entra al Expert mode.
  2. Ejecuta cpstop para detener los servicios.
  3. Elimina los archivos que indican que este servidor es Secondary: los archivos $FWDIR/state/ha_secondary*. Al eliminarlos, el servidor arrancará como Primary.
  4. Verifica que tienes una licencia mgmtha válida en este servidor. Recuerda: las licencias están vinculadas a la IP — si cambias la IP, la licencia queda inválida.
  5. Ejecuta cpstart. El servidor arranca ahora como Primary Active.
  6. Conecta SmartConsole al nuevo Primary. Elimina todas las instancias del objeto del Management antiguo (clic derecho → "Where Used" para encontrar todas).
  7. Instala la base de datos: Menu → Policy → Install Database → Selecciona todos los objetos → Install.
  8. Instala un nuevo Secondary con la IP del antiguo Primary (para mantener las licencias).
  9. Resetea SIC del nuevo Secondary y establece comunicación con el Primary.
🚨 Las licencias están vinculadas a la IP
Este es el punto más crítico del Disaster Recovery. Al final del proceso, verifica que todas las licencias tienen asignadas las IPs correctas. Si el nuevo Primary tiene una IP diferente al antiguo, las licencias deberán ser reemitidas desde el portal de licencias de Check Point (UserCenter).

Automatización con mgmt_cli y API REST

Check Point ofrece una API completa para gestionar la plataforma programáticamente. Desde crear objetos masivos hasta instalar políticas en scripts de CI/CD — la API es esencial para entornos de escala empresarial.

Herramientas de automatización disponibles

⌨️
mgmt_cli
Herramienta CLI standalone incluida en Gaia y en SmartConsole (Windows). Ideal para scripts de shell y Python. Disponible en Expert mode del Management.
🌐
Web Services (REST API)
HTTPS directo al Management Server. Compatible con Postman, curl, cualquier lenguaje. URL: https://<MGMT>/web_api/<comando>
🔑

Token estático para automatización sin credenciales de usuario. Se crea en SmartConsole → Manage & Settings → Administrators → API Key.

Librería oficial cpapi para Python. Simplifica la interacción con la API. Disponible en GitHub (CheckPointSW/cp_mgmt_api_python_sdk).

Configurar el API Server

El API Server debe estar habilitado y configurado para aceptar conexiones:

  1. En SmartConsole → Manage & Settings → Blades → Management API → Advanced Settings.
  2. En Startup Settings: selecciona "Automatic start" para que el API Server arranque con el Management (requiere más de 4 GB de RAM para habilitarse automáticamente).
  3. En Access Settings: selecciona quién puede conectarse:
    • Management server only: solo mgmt_cli local en el servidor
    • All IP addresses that can be used for GUI clients: los Trusted Clients definidos en SmartConsole
    • All IP addresses: cualquier IP puede usar la API
  4. Click OK. Publica los cambios.

El protocolo de escalamiento — Regla de Oro

Antes de llegar al kernel debug, siempre recorre estos niveles en orden. El 90% de los problemas se resuelven antes de llegar al Nivel 3:

✓ fw ctl zdebug + drop → identificar la función que descarta

Compliance Blade — Qué hace

La guía oficial describe el Compliance Blade como una solución dinámica que monitorea continuamente la infraestructura Check Point usando la tecnología CCM (Continuous Compliance Monitoring). Examina Gateways, Software Blades, políticas y configuraciones contra una base de datos extensa de estándares y mejores prácticas.

Cuándo escanea automáticamente

TriggerDescripción
Daily (diario)Un escaneo automático por día para detectar cambios en configuración de gateway o política hechos vía CLI o scripts (que no pasan por SmartConsole).
SmartConsole changesEscaneo automático cuando un administrador modifica objetos que afectan la configuración del Gateway o la política. Ocurre después del Publish.
ManualEl administrador puede lanzar un rescan en cualquier momento: Manage & Settings → Blades → Compliance → Settings → Rescan.

El sistema de scoring de Compliance

Cada best practice recibe un score numérico. El Compliance Blade calcula la media de todos los tests para obtener un score global:

EstadoScore (%)Acción recomendada
Poor0 – 50⚠️ Resolver INMEDIATAMENTE. 0 = completamente no-conforme.
Medium50 – 75Planificar corrección a corto plazo.
Good75 – 99Monitorear y mejorar gradualmente.
Secure100✅ Totalmente conforme con esta best practice.
N/A—El blade no está instalado o el Gateway no soporta la función evaluada.

Las 5 vistas del dashboard de Compliance

  1. Security Best Practices: lista de todas las best practices con su estado (Poor/Medium/Good/Secure). Click en cualquiera para ver: Description, Action Item (qué hay que hacer para corregir), y Relevant Objects (qué Gateways están afectados).
  2. Gateways: los 5 Gateways con mayor/menor score de compliance. Click en uno para ver las best practices específicas de ese Gateway.
  3. Blades: score promedio por Software Blade. Ver qué blades tienen las peores configuraciones.
  4. Action Items and Messages: lista priorizada de acciones correctivas pendientes.
  5. Regulatory Compliance: cumplimiento contra estándares específicos (PCI-DSS, ISO 27001, NIST, HIPAA, etc.).

Habilitar el Compliance Blade

  1. En SmartConsole → Gateways & Servers → doble click en el Security Management Server (no en el Gateway).
  2. En General Properties → pestaña Management → marca "Compliance".
  3. Click OK. Publica los cambios.
  4. Para ver el dashboard: Logs & Monitor → + (nuevo tab) → click "Compliance".

Best Practices personalizadas

Puedes crear tus propias best practices para requisitos organizacionales específicos:

Best Practice basada en regla de firewall

  1. Compliance tab → Security Best Practices → See All → New → Firewall Best Practice.
  2. Define nombre, descripción y Action Item (qué debe hacer el admin si se viola).
  3. En Best Practice Rule Definition: define los criterios de matching (Source, Destination, Service, Action, Track).
  4. En Violation Definition: selecciona "Rule not found" para verificar que existe una regla específica, o "Rule found" para verificar que NO existe una regla problemática.
  5. Configura el Hit Count como criterio (ej. alertar si una regla crítica tiene Hit Count = 0).

Best Practice de Gaia OS

  1. New → Gaia OS Best Practice.
  2. Define el script que se ejecutará en los Gateways (bash/shell).
  3. Define el "Expected Output" — si la salida del script coincide exactamente, el estado es "Secure".
  4. Ejemplo: script que verifica que SNMPv1/v2 está deshabilitado. Expected output: "SNMPv3 only".
